## Sprint Overview

This is a hardening sprint, not a feature sprint. The product exists. The goal is to make it work reliably in a 700-seat lecture theatre with 500+ students on poor network and low-end Android devices. Every task below serves that single goal.

## Priority 2 — GPS & Geofence Hardening (Backend Agent + CTO review)
**Deadline: May 25th, 2026 EOD**
Two students checked in during Test Run 2. We need 500 to check in reliably.

- [ ] Review current geofence radius setting — tune for lecture theatre size (~700 seats, estimate ~400sqm floor area)
- [ ] Add tolerance buffer for GPS drift on cheap Android devices (suggest ±15–20m buffer beyond physical geofence boundary)
- [ ] Test and verify check-in flow under simulated poor network (throttle to 2G/3G equivalent)
- [ ] Verify spoofing detection does not false-positive on legitimate weak GPS signals
- [ ] Confirm manual attendance fallback works end to end — lecturer can manually mark a student present
- [ ] Load test check-in endpoint for 500 concurrent requests — identify and fix any bottlenecks
- [ ] Add check-in idempotency — student cannot double-check-in for same session

## Demo Day Protocol — May 30th, 2026

- Arrive at venue early. Open the web app immediately to trigger server wake-up.
- Have 3–5 test accounts pre-created and pre-logged-in on your own devices as backup
- Have a printed manual attendance sheet as last-resort fallback
- Have the lecturer create the session before students enter the room
- Monitor Render dashboard on your phone during check-in window
